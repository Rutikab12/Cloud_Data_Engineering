2.--You have two tables:
orders(order_id, customer_id)
customers(customer_id, country)
Write a query to find total number of orders per country, including customers who have zero orders.
-
		“To include customers with zero orders, I will use customers as the left table and left join orders. Then I group by country and count order IDs.”
		SELECT
		  c.country,
		  COUNT(o.order_id) AS total_orders
		FROM customers c
		LEFT JOIN orders o
		ON c.customer_id = o.customer_id
		GROUP BY c.country;

		“COUNT(*) counts all rows, including rows where columns contain NULL values.

9.--Why did you choose Dataflow instead of PySpark for your pipelines?
		“We chose Dataflow because it is a fully managed, autoscaling service that supports both batch and streaming pipelines using a unified Apache Beam model.
		With PySpark, we would need to manage Dataproc clusters, handle scaling, and maintain separate code paths for batch and streaming.
		Dataflow reduced operational overhead and integrated better with GCP services like BigQuery and Pub/Sub.”
		
10.--You have a partitioned and clustered BigQuery table, but queries are still slow and expensive.
Give three concrete reasons why this might happen.
		“Even if a table is partitioned and clustered, queries can still be slow and expensive for several reasons:
		Partition filters are not used, causing BigQuery to scan all partitions
		Clustering columns are not used in filter or join conditions, so clustering provides no benefit
		Query patterns like SELECT *, unnecessary joins, or non-selective filters cause excessive data scanning
		Additionally, data skew or very high-cardinality clustering columns can also reduce clustering effectiveness.”

12.--Find the total order amount for customers who signed up in the last 30 days,
including customers who placed zero orders.
		SELECT
		  c.customer_id,
		  c.signup_date,
		  COALESCE(SUM(o.amount), 0) AS total_amount
		FROM bq_stage.customers c
		LEFT JOIN bq_stage.orders o
		ON c.customer_id = o.customer_id
		WHERE c.signup_date >= DATE_SUB(CURRENT_DATE(), INTERVAL 30 DAY)
		GROUP BY c.customer_id, c.signup_date;

13.--From a table transactions(user_id, txn_date, amount),
find users whose average transaction amount increased month-over-month.
	1.Aggregate transactions at user + month level
	2.Calculate monthly average amount
	3.Use LAG() to compare with previous month
	4.Filter users where average increased
		WITH monthly_avg AS (
		  SELECT
			user_id,
			DATE_TRUNC(txn_date, MONTH) AS txn_month,
			AVG(amount) AS avg_amount
		  FROM bq_stage.transactions
		  GROUP BY user_id, txn_month
		)
		SELECT
		  user_id,
		  txn_month,
		  avg_amount,
		  LAG(avg_amount) OVER (
			PARTITION BY user_id
			ORDER BY txn_month
		  ) AS prev_avg
		FROM monthly_avg
		QUALIFY avg_amount > prev_avg;

14.--Write logic (code or steps) to find users who had more than 3 events within any 10-minute window.
	import pandas as pd
	df=pd.read_csv('input.csv')
	df['event_time'] = pd.to_datetime(df['event_time'])
	window_counts = (
		df.groupby(
			['user_id', pd.Grouper(key='event_time', freq='10min')]
		)
		.size()
		.reset_index(name='event_count')
	)
	result = window_counts[window_counts['event_count'] > 3]

15.--An Airflow DAG triggers a streaming Dataflow job.
The DAG succeeds, but after 2 hours the Dataflow job crashes.
How do you detect this automatically and what actions do you take?
	“For streaming Dataflow jobs, Airflow success only indicates job submission, not job health.
	To detect crashes automatically, we rely on Cloud Monitoring alerts on Dataflow job metrics such as job state changes, worker failures, or zero throughput for a threshold duration.
	When an alert is triggered:
	The incident is logged and notified via email or PagerDuty
	The streaming job is restarted using a controlled deployment process
	Since the pipeline is idempotent, restarting does not cause data duplication
	Airflow is used only for orchestration; streaming job health is monitored independently.”
	
16.--Give two scenarios where Dataflow is a bad choice and explain what you would use instead.
		“Dataflow is not a good choice when the use case does not justify its complexity or cost.

		First, for small to medium batch workloads with simple transformations, Dataflow can be overkill. In such cases, I would use BigQuery SQL or Data Fusion, which are faster to develop and easier to maintain.
		Second, when transformations are purely SQL-based and data already resides in BigQuery, using BigQuery views, stored procedures, or scheduled queries is more efficient than running a Dataflow job.
		Dataflow is best suited for large-scale, complex, or streaming pipelines, not simple ELT workloads.”

25.--From employee(emp_id, dept, salary),find the second highest salary in each department.
		SELECT *
		FROM (
		  SELECT
			emp_id,
			dept,
			salary,
			DENSE_RANK() OVER (
			  PARTITION BY dept
			  ORDER BY salary DESC
			) AS rnk
		  FROM employee
		)
		WHERE rnk = 2;

26.--From orders(customer_id, order_date, amount),
--find customers whose latest order amount is greater than their average order amount.
		SELECT customer_id
		FROM (
		  SELECT
			customer_id,
			amount,
			ROW_NUMBER() OVER (
			  PARTITION BY customer_id
			  ORDER BY order_date DESC
			) AS rn,
			AVG(amount) OVER (
			  PARTITION BY customer_id
			) AS avg_amount
		  FROM orders
		)
		WHERE rn = 1
		  AND amount > avg_amount;
